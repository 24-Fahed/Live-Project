# 面向面试官的架构说明

> 这份文档面向面试交流，重点说明我在一个多上下文项目里，如何通过架构划分来降低 AI 协作和项目演进的复杂度。

## 1. 为什么要这样拆分项目

这个项目本身同时包含了：

- 微信小程序前端
- 管理后台前端静态资源
- FastAPI 网关
- WebSocket 实时通信
- HLS 流媒体访问链路
- 直播、投票、辩题、用户、AI 内容、统计等业务功能

如果把这些内容全部混在一起开发，不管是人工开发还是借助 AI 协作，都会很快遇到一个共同问题：上下文过大。上下文一旦过大，开发时就容易出现以下情况：

- 修改一个功能时误伤另一个功能
- AI 在回答或生成代码时混淆不同模块职责
- 难以快速定位问题边界
- 新功能加入后，原有结构越来越难维护

因此，我在这个项目里采用的核心思路不是“先堆功能”，而是“先把问题拆成有限上下文块，再在块内做实现”。

## 2. 我是如何把大项目拆成有限上下文块的

### 2.1 先按系统关注点拆成子系统

我先从系统层面做一级划分，把职责差异明显的部分拆成不同子系统：

- `frontend/`：微信小程序前端，只关心用户侧交互和页面逻辑
- `gateway/app/subsystems/wechat/`：负责微信登录接入
- `gateway/app/subsystems/stream/`：负责直播辩论核心业务
- `gateway/app/comm/ws/`：负责 WebSocket 通信与连接管理
- `gateway/app/infra/auth/`：负责统一鉴权
- `gateway/app/infra/static/`：负责静态资源挂载
- `gateway/app/subsystems/hls/`：早期用于演示 HLS 能力
- `gateway/app/subsystems/media/`：后续正式媒体层子系统

这样做的目标是让每个子系统先有明确边界，再避免“一个目录里既写用户界面、又写业务规则、又写基础设施”。

### 2.2 在子系统内部继续按业务能力拆成子模块

例如 `stream` 子系统并没有写成一个大文件，而是继续拆成：

- `live`：直播流管理与直播控制
- `vote`：投票逻辑
- `debate`：辩题管理
- `debate_flow`：辩论流程控制
- `judge`：评委管理
- `user`：用户列表与在线状态
- `dashboard`：管理端聚合视图
- `statistics`：统计分析
- `ai_content`：AI 内容相关能力
- `repository`：统一数据访问抽象

这样一来，每次开发、调试、补功能时，都可以先进入一个比较小的上下文块，而不是把整个项目一起装进脑子里。

## 3. 子系统之间如何保持解耦

我的处理原则是：子系统之间尽量通过清晰接口协作，而不是直接相互侵入实现细节。

具体来说：

- 微信登录相关逻辑集中在 `wechat` 子系统，不把微信登录细节散落到各个业务模块。
- 管理后台静态资源由 `gateway` 托管，但它不反向侵入业务实现，只通过 API 和 WebSocket 获取数据。
- WebSocket 通信抽成独立的 `ConnectionManager`，业务模块只关心“广播什么数据”，而不关心底层连接池如何维护。
- 鉴权放进中间件处理，业务模块不需要重复编写认证流程。
- 媒体访问链路单独处理，不让直播业务逻辑直接承担静态流媒体服务职责。

这种解耦方式的好处是：当我和 AI 一起工作时，我可以把任务描述得更小、更准，例如“只修改投票模块，不动登录和 WebSocket 管理器”，这样生成结果会更稳定。

## 4. 子模块之间保持什么程度的耦合

我并不是追求“完全零耦合”，因为课程项目和原型系统更看重可实现性与可维护性的平衡。

我采用的是“有限耦合、明确耦合点”的方式：

- 同一个子系统内部允许模块围绕统一数据抽象协作，例如 `stream` 各模块共享 `repository`。
- 聚合类模块可以依赖多个业务模块，例如 `dashboard` 读取 `live`、`vote`、`debate` 的结果，但它主要做聚合，不反向写入复杂业务规则。
- 广播类能力通过统一组件处理，例如 `vote` 和 `live` 都调用 `ws_manager.broadcast()`，但不各自维护一套 WebSocket 连接。
- 子模块之间尽量通过 service 层交互，而不是直接跨模块改底层数据结构。

换句话说，我允许必要的协作耦合，但不允许无序耦合。

## 5. 模块内部如何保持高内聚

为了让每个模块成为“对 AI 和开发者都友好的有限上下文块”，我尽量让模块内部保持高内聚：

- `router.py` 只负责接口入口和请求分发
- `service.py` 只负责业务逻辑
- `models.py` 只负责请求/响应模型
- `repository` 负责数据访问与存储抽象

这种结构有两个直接好处：

- 读代码时路径很稳定，容易找到问题落点
- 给 AI 下任务时，可以明确限定修改范围，例如“只改 `vote/service.py` 的投票计算，不改 router 和 repository”

从协作效率上看，这种结构可以有效减少 AI 因为上下文过大而产生的混乱，也能减少人工维护时的认知负担。

## 6. 在当前架构下如何扩展项目功能

当前架构最适合做“增量扩展”，也就是在不推翻现有结构的前提下加入新能力。

### 6.1 扩展一个新的业务模块

如果要在 `stream` 子系统下新增一个功能模块，典型做法是：

1. 在 `gateway/app/subsystems/stream/` 下新增一个模块目录
2. 新建 `models.py`、`service.py`、`router.py`
3. 如有数据需求，在 `repository` 中补充对应访问方法
4. 在 `stream/__init__.py` 中注册新 router
5. 如果需要实时同步，再复用现有 WebSocket 广播机制

这意味着增加功能通常是在局部完成的，而不是要求大面积改动。

### 6.2 扩展一个新的基础设施能力

如果未来要扩展例如：

- 新的鉴权方式
- 新的静态资源挂载
- 新的消息推送方式
- 数据库替换 `MockRepository`

也可以沿着现有基础设施分层做替换或增强，而不必把业务模块全部重写。

### 6.3 扩展到更大规模的系统

如果项目未来继续变大，当前架构还能继续演进：

- 先保持目录级模块化单体
- 再把数据访问抽象稳定下来
- 最后把高变化或高负载模块拆成独立服务

也就是说，当前结构既支持课程项目阶段的快速实现，也保留了后续演进空间。

## 7. 踩坑经验

### 7.1 微信小程序模拟器中的播放器不稳定

在这个项目里，一个非常实际的问题是：微信小程序模拟器里的播放器表现并不稳定。

因此在排查直播播放问题时，我不会只依赖模拟器现象来判断代码对错，而是会把“模拟器本身不稳定”作为排查前提之一。对于涉及视频播放、HLS 加载、页面切换后的状态恢复等问题，真机验证更有参考价值。

### 7.2 微信开发者工具本身问题较多

微信开发者工具在实际开发中有时会出现一些并不完全由业务代码导致的异常，例如：

- 页面状态不同步
- 缓存残留导致现象和代码不一致
- 编译后表现异常但重新编译又恢复正常

我的经验是：遇到异常现象时，不要立刻把所有问题都归结到业务代码上，应该优先尝试以下动作：

- 刷新项目
- 重新编译
- 重启微信开发者工具
- 必要时重新登录开发工具账号

### 7.3 微信账号必须登录

这一点看起来很基础，但实际很容易卡人：微信小程序相关调试依赖微信账号登录状态，如果开发者工具未登录，或者登录状态异常，很多现象都会失真，甚至直接影响接口测试和页面行为判断。

所以在排查小程序问题前，我通常会先确认：开发工具是否已经正确登录微信账号。

### 7.4 Windows 文件编码和 Linux 容器兼容问题

项目进入 Docker 调试后，我还踩到过一个很典型但容易忽略的问题：Windows 本地编辑出来的脚本和配置文件，进入 Linux 容器后可能因为编码和换行格式不兼容，直接导致服务启动失败。

我实际遇到过两类错误：

- `entrypoint.sh` 报 `exec format error`
- SRS 配置文件报 `illegal directive ﻿listen`

最后排查出来的根因不是业务代码本身，而是文件被保存成了：

- `UTF-8 BOM`
- `CRLF`

Linux 容器对这类文件很敏感：

- shell 脚本的 shebang 前如果混入 BOM，脚本可能无法执行
- SRS 配置文件第一行如果带 BOM，SRS 会把它当成非法指令的一部分

这次踩坑让我意识到，跨 Windows 开发环境和 Linux 容器环境时，部署问题不只是“代码逻辑对不对”，还包括“文件格式对不对”。后面我专门补了换行规范和 `.gitattributes`，把这类问题提前收敛掉。

### 7.5 媒体代理路径必须和 m3u8 内部路径一致

另一个非常实际的坑出现在媒体代理层。

最开始网关同时存在 `/media/...` 和 `/live/...` 两种媒体访问路径，但 SRS 返回的 `m3u8` 文件内部引用的是 `/live/...`。结果就是：

- 第一个播放地址可以访问
- 播放器继续请求 `m3u8` 内部引用的 `/live/...`
- 如果网关代理规则和实际引用路径不一致，就会出现 404

这个问题说明，媒体层接入不能只看“第一个 m3u8 地址能不能打开”，还要检查：

- `m3u8` 内部引用的路径是什么
- 分片、子清单、上下文参数是否还能继续走通

后来我把直播播放路径统一收敛成一种路径，并让网关代理规则和 SRS 输出路径保持一致，这个问题才真正解决。

## 8. 我想表达的核心能力

如果从面试角度总结，这个项目体现的不是“我写了很多功能”，而是：

- 我知道如何在复杂项目里先做结构化拆分
- 我知道如何控制模块边界，而不是让代码自然失控
- 我知道如何把项目拆成适合 AI 协作的有限上下文块
- 我知道怎样在当前阶段兼顾可实现性、可维护性和后续扩展性

对我来说，这种架构思维比单纯堆功能更重要，因为它决定了项目后面还能不能继续稳定演进。
