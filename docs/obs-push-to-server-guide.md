# OBS 推流到服务器教程

> 本文档说明如何使用 OBS 将直播流推送到本项目的媒体服务器。
> 当前约定方案：使用 `RTMP` 推流，媒体服务器为 `SRS`，OBS 可以直接推送到 `IP:PORT`。

## 1. 先理解推流目标

在当前方案下，OBS 负责把主播画面推到服务器。

最简单的推流目标是：

```text
rtmp://服务器IP:1935/live
```

其中：

- `服务器IP`：你的公网服务器 IP，或者本地调试时的本机 / 局域网 IP
- `1935`：RTMP 默认推流端口
- `live`：SRS 中配置的应用名

你还需要一个“流名称”，例如：

```text
stream-001
```

最终在 OBS 中通常要填两部分：

- 服务器：`rtmp://服务器IP:1935/live`
- 串流密钥：`stream-001`

## 2. 推流前需要准备什么

开始前请确认：

- Docker 已启动
- `gateway + srs` 容器已启动
- 服务器已经开放 RTMP 端口 `1935`
- 如果你是公网服务器联调，服务器防火墙已放行 `1935`

本地开发环境启动命令：

```bash
cd docker
docker compose --env-file .env.local -f docker-compose.yml -f docker-compose.local.yml up -d --build
```

服务器联调环境启动命令：

```bash
cd docker
docker compose --env-file .env.staging -f docker-compose.yml -f docker-compose.staging.yml up -d --build
```

正式上线环境启动命令：

```bash
cd docker
docker compose --env-file .env.prod -f docker-compose.yml -f docker-compose.prod.yml up -d --build
```

## 3. 本地调试时如何推流

如果你先在自己电脑上调试，最简单方式是：

### 3.1 查看本地推流地址

当前默认本地推流地址是：

```text
rtmp://127.0.0.1:1935/live
```

如果 OBS 和 Docker 在同一台机器上，这个地址通常就可以直接用。

如果你要让局域网内其他设备推流，可以改用你的局域网 IP，例如：

```text
rtmp://192.168.1.10:1935/live
```

### 3.2 OBS 中如何填写

打开 OBS：

1. 进入“设置”
2. 打开“推流”
3. 服务类型选择“自定义”
4. 服务器填写：

```text
rtmp://127.0.0.1:1935/live
```

5. 串流密钥填写：

```text
stream-001
```

### 3.3 开始推流后如何验证

推流开始后，你可以验证：

- 管理端接口：

```text
http://127.0.0.1:8080/api/v1/admin/media/streams/stream-001
```

- 播放地址：

```text
http://127.0.0.1:8080/live/stream-001.m3u8
```

## 4. 推流到公网服务器时怎么做

如果你已经把服务部署到公网服务器，最简单的方式是直接用公网 IP 推流。

### 4.1 假设你的公网 IP 是

```text
123.45.67.89
```

那么 OBS 中填写：

- 服务器：

```text
rtmp://123.45.67.89:1935/live
```

- 串流密钥：

```text
stream-001
```

### 4.2 推流后的播放地址

如果当前是公网 IP 联调阶段，播放地址通常是：

```text
http://123.45.67.89:8080/live/stream-001.m3u8
```

如果已经切到正式域名阶段，播放地址可能是：

```text
http://test.com:8080/live/stream-001.m3u8
```

注意：

- 当前用户访问可以走域名
- 但为了最简单上线，OBS 仍然可以继续走公网 IP 推流

## 5. 如果以后想用域名推流

当前阶段为了简单，建议 OBS 继续用公网 IP 推流。

如果后续你想改成域名推流，形式通常会变成：

```text
rtmp://live.test.com:1935/live
```

但这不是当前版本必须做的事情。

## 6. 你在业务上应该如何理解“开始直播”

你前面已经确认了一个业务规则：

- Admin 设置完整的 m3u8 地址
- 点击“开始直播”后，业务上就认为“该直播可以播放”
- 不强依赖系统在这一刻去验证媒体是否真的可播放

这意味着在当前项目里，OBS 推流和业务“开始直播”可以并行理解：

- 业务层负责把直播状态切到开始
- 媒体层负责真正提供流
- 小程序进入直播间后，根据地址去拉取资源并播放

所以从工程实现上说：

- 业务开始直播不需要等待 OBS 推流验证通过
- 只要播放地址最终可访问，用户侧就能正常拉流

## 7. 最常见的排查方法

### 7.1 OBS 无法连接服务器

优先检查：

- Docker 是否启动
- SRS 容器是否启动
- `1935` 端口是否映射成功
- 服务器防火墙是否放行 `1935`
- OBS 服务器地址是否写错

### 7.2 推流成功但播放地址打不开

优先检查：

- Gateway 是否启动
- SRS 是否已经生成 `.m3u8` 文件
- 播放地址路径是否正确
- 网关 `/live/*` 代理是否正常
- 使用的流名称是否和 OBS 推流密钥一致

### 7.3 本地能推，公网不能推

优先检查：

- 服务器安全组是否开放 `1935`
- 服务器本机防火墙是否开放 `1935`
- Docker 是否正确映射 `1935:1935`
- OBS 是否真的在推公网 IP，而不是本地地址

## 8. 一个完整示例

假设：

- 公网 IP：`123.45.67.89`
- 网关端口：`8080`
- RTMP 端口：`1935`
- 流名：`stream-001`

那么：

### 8.1 OBS 填写

服务器：

```text
rtmp://123.45.67.89:1935/live
```

串流密钥：

```text
stream-001
```

### 8.2 用户播放地址

```text
http://123.45.67.89:8080/live/stream-001.m3u8
```

### 8.3 如果已经有域名

用户播放地址可以变成：

```text
http://test.com:8080/live/stream-001.m3u8
```

而 OBS 仍然可以继续使用：

```text
rtmp://123.45.67.89:1935/live
```

## 9. 当前版本的最简单做法

如果你现在只追求“先推得上去、先播得出来”，那就按下面做：

1. 启动 Docker
2. 启动 `gateway + srs`
3. OBS 填：

```text
rtmp://服务器IP:1935/live
```

4. 串流密钥填：

```text
stream-001
```

5. 播放地址访问：

```text
http://服务器IP:8080/live/stream-001.m3u8
```

这就是当前版本最简单、最容易成功的推流方案。
