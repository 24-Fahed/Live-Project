# v0.1.2 版本规划草案

> 版本定位：完成 HTTPS 接入能力的版本  
> 文档性质：阶段性规划草案  
> 适用范围：Live-Project-Recover 当前版本后续演进

## 1. 版本目标

`v0.1.2` 定位为：**完成 HTTPS 接入能力的版本**。

本版本不以新增大型业务功能为核心目标，而是围绕系统的正式接入能力进行建设，使当前项目从“可本地运行、可公网 HTTP 调试”推进到“具备正式 HTTPS 接入能力”的状态。

本阶段的核心目标如下：

1. 完成系统的 HTTPS 接入能力设计。
2. 明确 HTTPS 属于基础设施层能力，而不是业务子系统。
3. 以全 HTTPS 作为 `prod` 环境目标。
4. 将 `staging` 定位为公网集成测试环境，按能力增量逐步开启配置。
5. 同步评估并纳入 `wss`（WebSocket over TLS）接入问题。
6. 新增 RESTful 风格 API 说明文档，完善工程文档体系。

## 2. 架构决策

### 2.1 HTTPS 的架构归属

HTTPS 属于**网关基础设施层的安全接入能力**，不作为独立业务子系统设计。

原因如下：
- HTTPS 不承载业务规则。
- HTTPS 解决的是安全接入、证书、传输加密、统一入口治理问题。
- HTTPS 为 `wechat`、`stream`、`admin`、`/live` 等上层能力提供底层支撑。

因此，在架构上应将其归入 gateway 的基础设施层，而不是新建业务子系统。

### 2.2 Cloudflare 的角色

Cloudflare 在整个 HTTPS 链路中的角色定义如下：

1. DNS 解析层
2. 公网侧 TLS / HTTPS 接入层
3. 反向代理层
4. 基础安全防护层

在 `v0.1.2` 中，Cloudflare 负责公网侧 HTTPS 接入，gateway 负责源站 HTTPS 服务。

### 2.3 源站通信模式

采用 Cloudflare `Full (strict)` 模式。

即：
- 客户端 -> Cloudflare：HTTPS
- Cloudflare -> Gateway：HTTPS

本阶段不采用“Cloudflare HTTPS，源站 HTTP”的降级模式。

## 3. 环境策略决策

### 3.1 环境分工

当前环境继续维持三阶段：

- `local`
- `staging`
- `prod`

各环境职责如下：

| 环境 | 定位 | 核心目标 |
| --- | --- | --- |
| local | 本地开发环境 | 以业务能力联调与快速验证为主 |
| staging | 公网集成测试环境 | 按能力增量逐步验证公网 IP、域名、HTTPS、WSS 等接入能力 |
| prod | 正式部署环境 | 以域名 + 全 HTTPS 方式提供正式访问能力 |

### 3.2 staging 的测试策略

`staging` 被定义为公网集成测试环境，并采用“按能力增量逐步开启配置”的策略。

建议的验证顺序为：

1. **公网 IP + HTTP**
   - 关闭 HTTPS 开关
   - 关闭域名开关
   - 验证服务器端口、Docker 部署、gateway 与 SRS 是否可用

2. **域名 + HTTP**
   - 开启域名开关
   - 保持 HTTPS 关闭
   - 验证 Cloudflare DNS 与域名回源链路是否正常

3. **域名 + HTTPS**
   - 开启 HTTPS 开关
   - 接入源证书与私钥
   - 验证 HTTPS、Cloudflare 回源、HLS、API、Admin、WebSocket 是否正常

该策略的目的不是长期保留多种接入状态，而是通过阶段化测试拆分故障域，提高集成测试定位效率。

### 3.3 prod 的接入目标

`prod` 环境采用**全 HTTPS**方案，不保留长期的 HTTP + HTTPS 混搭形式。

`prod` 的正式目标为：
- API：HTTPS
- Admin：HTTPS
- `/live` 播放：HTTPS
- WebSocket：WSS

## 4. WebSocket 联动影响

### 4.1 基本判断

启用 HTTPS 后，WebSocket 将受到联动影响，因此 `wss` 必须纳入 `v0.1.2` 规划范围。

原因：
- 当页面运行在 HTTPS 环境时，WebSocket 通常应切换为 `wss://`
- 如果页面是 HTTPS，但仍请求 `ws://`，会在浏览器或运行环境中触发安全限制

### 4.2 架构归属

`wss` 不作为独立能力单独设计，而应视为 HTTPS 接入能力的联动组成部分。

因此：
- HTTPS 是基础设施层能力
- WSS 是 HTTPS 接入后对实时通信链路的自然延伸
- `v0.1.2` 必须同时考虑 HTTP -> HTTPS 与 WS -> WSS 的联动变化

### 4.3 规划要求

在本阶段计划中，必须把以下内容纳入测试范围：
- HTTPS 页面能否正常建立 WebSocket 连接
- WebSocket 地址生成逻辑是否与网关入口统一
- 前端运行模式是否能够平滑过渡到 WSS 方案

## 5. 配置策略决策

### 5.1 当前阶段的配置方式

当前阶段继续保留按部署场景划分的环境文件：

- `.env.local`
- `.env.staging`
- `.env.prod`

本阶段**不拆分为多个物理 env 文件**。

理由：
- 当前项目规模尚不足以支撑进一步拆分的复杂度
- 继续使用场景化 env 文件更利于调试与部署
- 可以先通过逻辑分组提升清晰度，再视后续规模决定是否物理拆分

### 5.2 文件内部的逻辑分组要求

每个环境文件内部应至少按以下类别分组：

1. 运行环境配置
2. 公网访问配置
3. HTTPS / TLS 配置
4. 媒体服务配置
5. 微信小程序配置
6. 安全令牌配置
7. Docker 镜像源 / pip 源配置

### 5.3 开关策略

本阶段采用“**能力保留、按环境启用**”的策略。

建议保留少量高层开关，例如：
- `HTTPS_ENABLED`
- `DOMAIN_ENABLED`
- `CLOUDFLARE_ENABLED`

其作用是：
- `local` 中默认关闭 HTTPS 与域名能力
- `staging` 中按测试阶段逐步开启
- `prod` 中完整开启

此策略用于降低本地开发复杂度，同时保证整体架构保持完整。

## 8. 软件工程原则

### 8.1 基础设施与业务分离

HTTPS / WSS 应被视为基础设施能力，不进入业务子系统边界。

### 8.2 低耦合原则

HTTPS 接入不得侵入 `stream`、`wechat` 等业务模块内部逻辑。

### 8.3 统一入口原则

`/api`、`/admin`、`/live` 以及 WebSocket 入口应继续由 gateway 统一管理。

### 8.4 渐进式集成测试原则

`staging` 环境按能力增量逐步启用配置，以降低问题定位成本。

### 8.5 配置清晰原则

本阶段不拆分物理 env 文件，但必须在场景化 env 文件内部完成严格分组。

### 8.6 最小改动原则

`v0.1.2` 聚焦 HTTPS 接入能力与相关联动能力，不引入超出本阶段目标的大规模业务重构。

`v0.1.2` 的核心任务不是继续扩展业务功能，而是完成系统从“原型可运行”向“具备正式接入能力”过渡所必须的基础设施建设。

本阶段的正式决策为：

1. HTTPS 归入基础设施层。
2. `staging` 作为公网集成测试环境，按能力增量逐步开启配置。
3. `prod` 采用全 HTTPS 方案。
4. WSS 作为 HTTPS 联动能力纳入本阶段范围。
5. RESTful 风格 API 文档纳入本阶段规划。
