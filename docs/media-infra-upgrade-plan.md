# 下版本基础设施改造依据

> 这份文档只讨论“为引入正式媒体层而需要修改的基础设施内容”，作为下一个版本开发时的实施依据。
> 文档目标：尽量简单、可落地、适合当前项目平滑升级。
> 已确认前提：不修改前端代码；媒体服务器选用 `SRS`；OBS 推流协议选用 `RTMP`；允许 OBS 直接使用 `IP:PORT` 推流，以满足最简单上线。

## 1. 本次改造的核心目标

当前项目最大的缺陷之一，是媒体层实际上缺失。

现有 `hls` 子系统本质上只是一个演示级方案，能勉强完成“视频播放演示”，但不能承担正式直播系统中的以下职责：

- 接收主播的 OBS 推流
- 进行媒体转封装与分发
- 提供稳定的播放地址
- 管理直播中的媒体状态

因此，下一个版本的目标不是“继续修补当前 hls 子系统”，而是：

- 删除现有演示性质很强的 `hls` 子系统
- 引入独立的正式媒体层子系统
- 使用 `SRS` 作为媒体服务器
- 让 `FastAPI Gateway` 继续作为业务网关
- 让网关负责 API、Admin、播放入口的统一组织
- 允许 OBS 通过独立 RTMP 端口直接推流到服务器

## 2. 本次改造的边界

### 2.1 不做的事情

为了保证下一个版本能尽快上线，这次改造明确不做以下事情：

- 不修改 `frontend/` 现有前端代码
- 不把 OBS 推流强行收敛到 `8080` 单端口
- 不继续使用 Cloudflare Tunnel
- 不让 FastAPI 自己处理专业媒体协议

### 2.2 要做的事情

- 用 Cloudflare DNS 做正式域名解析
- 使用 1 个公网 IP + 1 个域名完成上线
- API、Admin、播放相关 HTTP 入口尽量统一由网关组织
- OBS 推流走独立 RTMP 端口，例如 `1935`
- 允许 OBS 为了最简单上线，直接推到 `公网IP:1935`
- 在网关基础设施层补齐“媒体接入所需的基础能力”
- 将“本地开发 / 线上调试 / 正式上线”的地址切换能力做成 Docker 的配置能力，而不是改代码

## 3. 新增的部署与调试需求

这一部分是本次讨论新增的关键需求。

### 3.1 需求背景

当前系统一旦进入下个版本，就不再只是本地演示，而是要真正面对“本地调试、服务器联调、正式上线”三个阶段。

这会带来一个实际问题：

- 本地调试时通常使用局域网 IP 或本机 IP
- 本地阶段不走 DNS
- 服务器联调时开始使用公网 IP
- 正式上线后使用域名访问

如果这些切换依赖“改代码”“改常量”“手动改地址文件”，后期会非常混乱，也容易在上线时漏改配置。

### 3.2 正式需求表述

下一个版本必须支持通过 Docker 配置切换不同运行阶段所需的地址信息，包括：

- 本地开发阶段
- 服务器调试阶段
- 正式上线阶段

这些阶段的切换应通过以下手段完成：

- Docker Compose 文件
- 不同环境的 `.env` 文件
- 容器启动配置
- 网关基础设施配置层

而不应通过直接修改业务代码实现。

### 3.3 这个需求的目标

这个需求的目标是：

- 让地址切换成为部署能力，而不是编码行为
- 让本地与线上使用同一套代码
- 让不同运行阶段有清晰边界
- 让“调试到什么程度算完成、什么时候进入下一阶段”变得可判断

## 4. 最终访问方式

本次改造后的访问方式建议分成两类：

### 4.1 用户访问路径

- 浏览器 / 小程序 -> 域名 -> Cloudflare DNS -> 公网 IP -> Gateway `:8080`

这条路径用于：

- API
- Admin 页面
- 播放地址访问

### 4.2 主播推流路径

- OBS -> `公网IP:1935`

或者后续可以升级为：

- OBS -> `live.test.com:1935`

但为了最简单上线，当前版本允许直接使用 IP 推流。

## 5. 基础设施层需要改什么

当前 `gateway/app/infra/` 下只有：

- `auth/`
- `static/`

这说明目前基础设施层还偏弱，只覆盖了“鉴权”和“静态资源挂载”。

为了支持正式媒体层，下一个版本建议把 `infra/` 扩展为下面这种结构：

```text
gateway/app/infra/
├─ auth/
├─ static/
├─ runtime/
├─ media_proxy/
├─ media_config/
├─ edge/
└─ security/
```

注意：

- 这里的名字可以微调，但职责不要混。
- 重点不是目录名，而是“每个基础设施能力要独立出来，不要散落在业务代码中”。

## 6. 我建议新增的基础设施能力

### 6.1 `infra/runtime/`

这是本次讨论新增后非常重要的基础设施目录。

它负责描述“当前系统处于什么运行阶段”，例如：

- local
- staging
- production

建议新增文件：

```text
gateway/app/infra/runtime/
├─ __init__.py
└─ config.py
```

建议职责：

- 定义当前运行模式
- 定义当前是否使用域名
- 定义当前是否允许直接 IP 推流
- 定义当前播放地址生成策略
- 定义当前是否启用公网回调地址

这个目录的意义是：

- 让“本地 / 线上调试 / 正式上线”的差异进入基础设施层，而不是散落在业务模块里

### 6.2 `infra/media_config/`

这个目录负责管理媒体层相关的配置。

建议放的内容：

- `SRS` 地址
- `SRS` 的 RTMP 端口
- `SRS` 的 HTTP 播放端口
- 外部播放域名
- 推流默认应用名
- 播放默认路径前缀
- 是否允许公网 IP 推流
- 是否启用回调鉴权

建议新增文件：

```text
gateway/app/infra/media_config/
├─ __init__.py
└─ config.py
```

这个配置层的意义是：

- 以后切换测试环境、正式环境时，不要到处改硬编码
- 让媒体层接入信息统一从基础设施层读取

### 6.3 `infra/media_proxy/`

这个目录负责“播放入口接入”，也就是把用户侧访问的播放地址统一组织到网关下面。

它不负责处理 RTMP。
它负责的是 HTTP 类型的播放访问，例如后续的：

- HLS 播放路径代理
- 播放地址转发
- 播放地址重写
- 响应头控制

建议新增文件：

```text
gateway/app/infra/media_proxy/
├─ __init__.py
├─ config.py
└─ router.py
```

建议职责：

- `config.py`：播放代理路径配置
- `router.py`：统一处理 `/live/*` 播放访问入口，并把请求转发到 SRS

这样做的好处是：

- 用户侧还是通过网关访问媒体播放地址
- 业务系统不需要直接暴露 SRS 内部地址
- 将来不同阶段只需要改代理配置，不需要改代码

### 6.4 `infra/edge/`

这个目录负责“入口层规则”，也就是统一定义网关对外暴露时的接入约束。

建议新增文件：

```text
gateway/app/infra/edge/
├─ __init__.py
└─ config.py
```

建议职责：

- 定义系统对外统一入口端口信息
- 定义公网访问域名
- 定义是否运行在 Cloudflare DNS 场景下
- 定义反向代理/真实 IP 相关配置
- 定义播放域名与 API 域名是否共用

这个目录的意义是：

- 让“系统如何暴露到公网”成为明确配置，而不是散落在 Docker、脚本、代码里

### 6.5 `infra/security/`

这个目录负责媒体层接入后新增的一些基础安全控制。

建议新增文件：

```text
gateway/app/infra/security/
├─ __init__.py
└─ media_security.py
```

建议处理的内容：

- 限制 SRS 回调来源
- 校验回调鉴权 token
- 限制敏感管理接口访问
- 后续支持推流密钥校验

当前版本为了最简单上线，可以先只做最小安全：

- 回调 token
- 管理接口保护

## 7. 我建议修改的现有基础设施文件

### 7.1 修改 `gateway/app/infra/auth/config.py`

当前这个文件只考虑了普通 API 和静态资源白名单。

下一个版本中，它需要补充“媒体接入后的白名单与鉴权边界”。

建议修改方向：

- 增加媒体回调白名单路径
- 增加播放代理白名单路径
- 保持管理接口和普通业务接口的鉴权边界清晰

建议新增的思路：

- 播放地址一般允许匿名访问
- SRS 回调接口应走单独 token 校验，而不是普通用户 JWT
- 管理媒体层的接口仍走管理侧权限控制

### 7.2 修改 `gateway/app/infra/static/config.py`

这个文件现在只负责：

- `/admin`
- `/static`

下一个版本里它不应该再承担 HLS 服务职责，但要明确“静态资源层不再负责媒体服务”。

也就是说：

- 继续保留 admin 静态资源挂载
- 不再新增任何演示性质的媒体文件挂载
- 把媒体播放入口交给 `infra/media_proxy/`

### 7.3 修改 `gateway/app/infra/__init__.py`

当前这个文件几乎没有承担基础设施总入口职责。

下一个版本建议让它能统一导出基础设施能力，例如：

- auth
- static
- runtime
- media_config
- media_proxy
- edge
- security

这样以后主应用在 `main.py` 中初始化基础设施时会更清晰。

## 8. 我建议新增的环境变量

为了让下一个版本更容易部署，建议把下面这些变量纳入网关环境配置。

### 8.1 运行阶段相关

- `APP_ENV=local`
- `APP_RUNTIME_MODE=local`
- `USE_DOMAIN=false`
- `USE_PUBLIC_IP=false`

建议说明：

- `local`：本地开发
- `staging`：服务器联调
- `production`：正式上线

### 8.2 边缘接入相关

- `PUBLIC_DOMAIN=test.com`
- `PUBLIC_BASE_URL=http://test.com:8080`
- `LOCAL_BASE_URL=http://192.168.x.x:8080`
- `STAGING_BASE_URL=http://公网IP:8080`
- `USE_CLOUDFLARE_DNS=true`
- `GATEWAY_PORT=8080`

### 8.3 SRS 接入相关

- `SRS_HOST=127.0.0.1`
- `SRS_RTMP_PORT=1935`
- `SRS_HTTP_PORT=8088`
- `SRS_APP=live`
- `SRS_PLAY_PATH_PREFIX=/live`

- `LOCAL_PUSH_BASE=rtmp://192.168.x.x:1935/live`
- `STAGING_PUSH_BASE=rtmp://公网IP:1935/live`
- `PRODUCTION_PUSH_BASE=rtmp://公网IP:1935/live`

### 8.4 安全相关

- `SRS_CALLBACK_TOKEN=replace-me`
- `MEDIA_ADMIN_TOKEN=replace-me`

## 9. Docker 需要如何支持不同阶段切换

这是下一个版本必须实现的部署能力。

### 9.1 核心原则

不同运行阶段的切换必须通过 Docker 完成，而不是改代码。

建议用下面三类文件配合：

- `docker-compose.yml`
- `docker-compose.local.yml`
- `docker-compose.staging.yml`
- `docker-compose.prod.yml`
- `.env.local`
- `.env.staging`
- `.env.prod`

### 9.2 推荐切换方式

例如：

```bash
docker compose --env-file .env.local -f docker-compose.yml -f docker-compose.local.yml up -d
```

```bash
docker compose --env-file .env.staging -f docker-compose.yml -f docker-compose.staging.yml up -d
```

```bash
docker compose --env-file .env.prod -f docker-compose.yml -f docker-compose.prod.yml up -d
```

这个能力的意义是：

- 本地调试时自动使用局域网 IP
- 服务器联调时自动使用公网 IP
- 正式上线时自动使用域名
- 所有切换都在部署层完成，不改业务代码

## 10. 各阶段应当调试什么

### 10.1 本地开发阶段

本地开发阶段重点调试：

- 网关 API 是否正常
- Admin 页面是否正常
- SRS 是否能接收 OBS 推流
- 播放代理是否能正确返回播放内容
- 业务系统与媒体系统的基本联动是否正常

这个阶段通常使用：

- 局域网 IP
- 本机 Docker
- 不依赖 DNS

当以下条件成立时，可以进入服务器联调阶段：

- OBS 本地推流成功
- 播放地址本地可用
- 网关能正确返回推流地址和播放地址
- 媒体状态能被业务系统感知

### 10.2 服务器联调阶段

服务器联调阶段重点调试：

- Docker 部署是否能在服务器环境完整跑通
- 公网 IP 推流是否成功
- 公网 IP 播放是否成功
- 防火墙 / 端口映射是否正确
- SRS 回调是否能打到网关
- 管理端是否能正确看到流状态

这个阶段通常使用：

- 公网 IP
- 不一定先启用域名
- 重点检查网络与部署链路

当以下条件成立时，可以进入正式上线阶段：

- 公网 IP 推流稳定
- 公网 IP 播放稳定
- 管理端联动正常
- Docker 重启后服务可恢复
- 配置切换不需要改代码

### 10.3 正式上线阶段

正式上线阶段重点调试：

- 域名解析是否生效
- Cloudflare DNS 是否正常指向服务器
- 域名访问 API / Admin 是否正常
- 域名访问播放地址是否正常
- 线上回调、安全控制、日志记录是否正常

当以下条件成立时，可以认为“上线完成”：

- 域名访问稳定
- 推流和播放稳定
- Docker 部署可重复执行
- 关键功能在重启后仍能恢复
- 本地、联调、线上三种环境切换都只靠 Docker 配置完成

## 11. 本次版本推荐的最小上线方案

为了降低复杂度，我建议你把“下一个版本上线”理解成下面这个最小目标：

### 11.1 推流

- OBS 直接推到：`公网IP:1935`

这一步最简单，也最符合你“先上线”的目标。

### 11.2 播放

- 用户通过网关暴露的播放地址访问媒体内容
- 例如由网关提供：
  - `/live/{stream}.m3u8`
  - 或 `/play/{stream}`

至于网关背后最终转到 SRS 的什么内部地址，这个对前端和用户应该尽量隐藏。

### 11.3 API

- 管理端通过 FastAPI 获取推流地址、播放地址、直播状态
- 业务系统通过回调或轮询获知 SRS 的流状态变化

## 12. 本次基础设施改造的实施顺序

建议按下面顺序做，不要一次全改。

### 第一步：清理旧媒体演示方案

- 标记 `hls` 子系统为待删除
- 停止继续在它上面追加功能
- 把“媒体能力缺失”作为明确问题记录下来

### 第二步：补基础设施配置层

- 新增 `infra/runtime/`
- 新增 `infra/media_config/`
- 新增 `infra/edge/`
- 修改 `auth/config.py`

这一步的目标是先把“媒体接入需要哪些系统配置”和“不同运行阶段如何切换”整理好。

### 第三步：补播放入口层

- 新增 `infra/media_proxy/`
- 在网关中增加统一播放入口

这一步的目标是把用户访问入口先统一到网关下面。

### 第四步：补安全控制

- 新增 `infra/security/`
- 增加回调 token 校验

这一步的目标是避免媒体层一接入就完全裸奔。

## 13. 这份文档对下版本开发的意义

这份文档最重要的作用，不是直接指导你写每一行代码，而是先把下面这几件事定下来：

- 下一版本不再把“媒体播放演示”误当成“正式媒体层”
- 媒体能力必须是一个独立子系统
- 基础设施层要先补齐，不能让媒体接入逻辑散落在业务模块里
- 网关继续做业务入口，不替代专业媒体服务器
- OBS 推流允许先走独立 RTMP 端口，以保证最简单上线
- 不同阶段的 IP / 域名切换必须成为 Docker 配置能力，而不是代码修改行为

## 14. 最终建议

如果只给一个最实用的建议，那就是：

- 先接受“RTMP 推流端口独立”
- 先用 `SRS + OBS + Gateway` 跑通最小闭环
- 先把基础设施层补齐
- 先把运行阶段切换做成 Docker 能力
- 不要在下一个版本同时追求“零改前端、单端口统一、现代低延迟协议、最强安全、最完美架构”

对于你当前这个阶段，最正确的目标不是“一步到位”，而是：

- 先把正式媒体层接进来
- 让系统第一次真正具备直播能力
- 让部署切换不依赖改代码
- 然后再做后续的结构优化
