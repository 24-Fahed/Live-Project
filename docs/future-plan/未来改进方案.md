# 未来改进方案

> 本文档描述项目的下一阶段部署改进目标：
> 将系统的对外访问统一升级为 HTTPS，使用 Cloudflare 提供 TLS 能力，由 Gateway 提供统一的 HTTPS 入口，并继续接入媒体服务。

## 1. 目标

当前项目可以先以“HTTP 页面/API + HTTPS 播放”的方式完成最小上线，但从长期演进和正式部署角度看，更合理的方向是：

- 页面访问使用 HTTPS
- API 访问使用 HTTPS
- Admin 访问使用 HTTPS
- 小程序播放地址使用 HTTPS
- 系统对外统一通过域名访问
- Cloudflare 负责外部 TLS 接入
- Gateway 负责系统内部统一入口与业务转发

最终目标是把系统升级为“全站 HTTPS”的部署模式。

## 2. 总体方案

本方案的核心思想是：

- Cloudflare 作为外部访问入口，负责客户端到 Cloudflare 之间的 TLS
- Gateway 作为系统统一入口，负责源站 HTTPS 服务能力
- SRS 作为媒体服务，继续负责 RTMP 接收和 HLS 输出
- 用户侧不再直接感知内部服务拆分，而是统一通过域名访问

也就是说，最终链路是：

- 浏览器 / 小程序 -> Cloudflare -> Gateway -> 内部服务

其中：

- Cloudflare 负责公网 HTTPS 接入
- Gateway 负责站内统一路由和媒体播放入口组织
- 内部服务之间可以根据实现复杂度选择 HTTP 或 HTTPS

## 3. 域名规划

建议在 Cloudflare 中至少配置以下 DNS 记录：

- `A test.com -> 公网 IP`
- `A live.test.com -> 公网 IP`

其中：

- `test.com` 用于主站、API、Admin
- `live.test.com` 用于直播播放相关访问

如果后续希望进一步简化，也可以继续收敛到单域名，但在当前阶段，主站域名和媒体域名分开更清晰、更便于排查。

## 4. HTTPS 能力分工

### 4.1 Cloudflare 负责什么

Cloudflare 负责：

- 用户侧 HTTPS 接入
- 域名解析
- TLS 证书管理
- 外部安全接入的第一层保护

这意味着用户访问时，看到的是：

- `https://test.com/api/...`
- `https://test.com/admin/...`
- `https://live.test.com/live/stream-001.m3u8`

### 4.2 Gateway 负责什么

Gateway 负责：

- 提供源站 HTTPS 服务能力
- 统一接收来自 Cloudflare 的请求
- 转发 API 请求到业务子系统
- 托管 Admin 前端静态资源
- 统一组织媒体播放入口，例如：
  - `/media/...`
  - `/live/...`
- 与媒体层联动，向前端返回播放地址和状态信息

这里强调“HTTPS 能力由网关提供”，意思不是 Cloudflare 不做 HTTPS，而是：

- Cloudflare 提供公网侧 TLS
- Gateway 也具备源站 HTTPS 能力
- 二者配合，形成完整 HTTPS 链路

### 4.3 媒体服务负责什么

媒体服务仍然由 SRS 承担，负责：

- 接收 OBS 的 RTMP 推流
- 输出 HLS 播放资源
- 配合 Gateway 提供对外播放地址

推流侧在当前阶段仍可以保持：

- `rtmp://公网IP:1935/live`

也就是说：

- 播放走 HTTPS
- 推流先保持 RTMP

这样可以降低整体改造风险。

## 5. 访问方式设计

### 5.1 用户访问

用户访问统一改为 HTTPS：

- `https://test.com/`
- `https://test.com/api/...`
- `https://test.com/admin/...`
- `https://live.test.com/live/stream-001.m3u8`

### 5.2 OBS 推流

OBS 推流在改进方案中继续保持最简单做法：

- `rtmp://公网IP:1935/live`

流名例如：

- `stream-001`

后续如果要进一步演进，可以再研究基于域名的推流入口，但不作为本方案的第一阶段目标。

## 6. Gateway 的职责变化

为了支持这个改进方案，Gateway 需要从“仅提供 HTTP 网关”升级为“提供 HTTPS 统一入口的网关”。

也就是说，Gateway 在下一阶段不仅要负责：

- API
- Admin
- WebSocket
- 媒体播放代理

还要负责：

- 源站证书加载
- HTTPS 端口监听
- 统一 HTTPS 路由组织

这一步完成后，Gateway 才真正成为系统的统一接入层。

## 7. Cloudflare 配置思路

当前你已经明确会在 Cloudflare 中添加：

- `A live.test.com -> 公网 IP`

在未来改进方案中，建议同时配置：

- `A test.com -> 公网 IP`
- `A live.test.com -> 公网 IP`

然后在 Cloudflare 的 SSL/TLS 页面中完成：

- 启用 HTTPS
- 配置源站证书策略
- 让 `test.com` 和 `live.test.com` 都进入 HTTPS 访问链路

这样主站和媒体域名都能通过 Cloudflare 的 TLS 能力暴露给外部。

## 8. 推荐的最终部署结构

建议形成如下结构：

- Cloudflare：负责外部 HTTPS
- Gateway：负责源站 HTTPS、API、Admin、播放入口
- SRS：负责媒体推流与分发
- Docker：负责不同环境的部署切换

最终逻辑上可以理解为：

- `test.com`：主业务域名
- `live.test.com`：播放域名
- `Gateway`：统一入口
- `SRS`：媒体处理

## 9. 为什么这是未来改进方案

因为这套方案比当前“最小上线方案”更完整，但也更重一些。

它适合作为未来改进方案，而不是当前第一步就强行全部落地，原因是：

- 需要补全网关 HTTPS 能力
- 需要补全证书与 HTTPS 配置
- 需要统一调整访问地址输出策略
- 需要完成 Cloudflare 到源站的 HTTPS 链路设计

因此，这份文档的定位是：

- 为下一阶段正式改造提供方向
- 明确全站 HTTPS 的最终目标
- 让后续开发有统一的部署思路

## 10. 结论

未来改进方向可以明确为：

- 全站统一使用 HTTPS
- Cloudflare 作为外部 TLS 能力提供方
- Gateway 作为系统统一 HTTPS 入口
- SRS 继续承担媒体处理能力
- 主站域名与媒体域名都接入 Cloudflare

在这个方案下，系统最终会从“可运行的直播原型”升级为“具备正式 HTTPS 接入能力的完整直播系统”。
