# 软件架构设计说明

## 一、软件特性

### 1. 架构风格

本系统采用基于 **FastAPI 的网关架构**，整体设计遵循 RESTful 风格，适用于前后端分离的 Web 应用及小程序场景。

在架构设计上，参考微服务架构思想，同时进行规模收敛与工程化简化，使系统既具备良好的扩展能力，又避免传统微服务带来的复杂性。

---

### 2. 分层与结构设计

系统采用分层结构：系统 → 子系统 → 模块 → 构件


设计原则如下：

- **子系统级扩展**
  - 通过新增子系统扩展整体系统功能
- **模块级扩展**
  - 通过新增模块扩展子系统能力
- **网关接入统一化**
  - 所有子系统通过 Router 统一接入网关
- **子系统内部结构清晰**
  - router：负责请求接入与路由分发
  - service：负责业务逻辑处理
  - models：负责数据模型定义

---

### 3. 解耦与协作原则

- 子系统之间通过接口通信，**禁止直接修改对方数据**
- 模块职责划分明确，降低耦合，便于团队协作
- 模块内部保持高内聚，子系统间保持低耦合
- 支持子系统独立开发、测试与部署，互不影响

---

### 4. 多服务器部署能力

系统具备灵活的部署能力：

- 网关提供统一基础设施层
- 子系统通过标准化接口接入网关
- 支持以下部署模式：单体部署 → 多实例部署 → 类微服务部署

具体表现为：

- 可复制整个项目至不同服务器
- 通过调整启动配置选择运行的子系统
- 每个子系统可独立运行于不同主机或端口
- 无需修改代码或项目结构即可完成扩展

---

### 5. 流媒体与安全

- 使用 HTTPS 提供安全访问
- 基于 Cloudflare Tunnel 实现临时 HTTPS 接入与流量保护
- 支持直播流的安全访问与加速

---

### 6. Docker部署

提供docker文件，通过docker compose构建/停起后端服务。

## 二、功能修改

## 五、已知问题与优化计划

### 1. 直播状态显示异常

问题描述：

- 新建直播间后：
  - 状态显示异常（显示为“开播”但实际未开播）
  - UI 标识错误（灰色标记与交互状态不一致）

- 已存在直播间：
  - 状态切换正常

---

### 2. 直播流更新未同步

问题描述：

- 创建或修改直播流后：
  - 小程序未接收到辩题推送
  - 点击“开始直播”无推送效果

---

### 3. 页面切换导致状态异常

问题描述：

- 切换标签页后返回：
  - 直播状态 UI 错误（绿色状态变为异常标识）

---

### 4. 用户在线状态不准确

问题描述：

- 用户登录后仍显示离线
- 进入/退出直播间状态变化异常

改进方向：

- 基于 WebSocket 注册状态判断用户是否在线
- 区分“系统在线”与“直播间在线”状态

---

### 5. AI 内容未同步到小程序

问题描述：

- 在 Admin 启动 AI 识别后：
  - 小程序端未显示 AI 内容

---

### 6. 分页功能异常

问题描述：

- AI 内容管理列表：
  - “下一页”按钮无效

---

### 7. UI 显示问题

问题描述：

- 部分符号缺失或渲染异常

---

### 8. WebSocket 注册逻辑问题

问题描述：

- 用户在选择界面时：
  - 未注册到直播间
  - 导致在线状态判断错误

---

### 9. HTTPS 方案（临时）

当前方案：

- 使用 Cloudflare Tunnel 提供 HTTPS 能力

后续计划：

- 域名申请完成后：
  - 切换至正式 HTTPS + 服务器部署方案

---

### 10. 小程序性能优化

问题描述：

- 小程序体积过大

优化方向：

- 分包加载
- 模块裁剪与资源优化

---

### 11. Admin 后台安全增强

计划：

- 限制后台访问 IP
- 提高系统安全性

---

## 六、总结

本系统通过“类微服务架构”实现：

- 高扩展性
- 低耦合设计
- 灵活部署能力

在保证开发效率的同时，兼顾系统的长期演进能力。当前阶段重点在于：

- 修复状态同步问题
- 优化前端表现
- 完善 HTTPS 与部署体系
